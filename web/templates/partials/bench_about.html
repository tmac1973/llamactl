What benchmarks measure

Two source paths produce results, both via real HTTP requests through the running router:

  • internal-* presets drive an in-process loop that sends chat completion requests with a known prose prompt sized to a target token count and reads the timings llama.cpp returns in the response body.
  • benchy-* presets shell out to llama-benchy via uvx, an external API benchmarker that supports concurrency and works with sharded GGUFs (where the legacy llama-bench tool fell over).
Internal prompt

Every internal request is built by repeating a fixed prose passage until it hits the target character count, prefixed with a per-rep header to defeat llama.cpp's prompt cache. Approximate sizing: {{.InternalPrompt.CharsPerToken}} chars/token (English BPE average).

Metrics
PP t/s (Prompt Processing tokens/sec)
How fast the model reads and processes input prompt tokens. GPU compute-bound; benefits from batching.
TG t/s (Token Generation tokens/sec)
How fast the model generates response tokens. Memory bandwidth-bound on GPU; this is the speed you feel during chat.
TTFT (Time To First Token, ms)
Wall-clock time from request to first response token. Includes prompt processing.
TTFR (Time To First Response, ms) — benchy only
llama-benchy's separately-timed first-response measurement, distinct from end-to-end TTFT.
peak t/s — benchy only
Peak observed throughput within the run, useful when concurrency > 1 to spot saturation.
e2e_ttft (end-to-end TTFT, ms) — benchy only
llama-benchy's end-to-end first-token timing, comparable to TTFT above.
Live performance

The Server tab's Live Performance panel passively captures timing data from every chat completion routed through the proxy — no explicit benchmark needed. It shows a rolling average of real-world performance across recent requests.
