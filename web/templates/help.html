Llama Toolchest — Help

A product manual for running local LLMs with the llama.cpp inference server, wrapped in a web UI.

Quickstart

  1. Build llama.cpp. Go to Builds, pick a profile that matches your hardware (CUDA, ROCm, CPU), and start a build. Builds are cached and reused.
  2. Download a model. On Search HF, search for a GGUF repo (e.g. unsloth/Qwen3.5-4B-GGUF) and download a quant (Q4_K_M is a good balance for most).
  3. Enable it. On Models, flip the "On" switch for the model(s) you want the server to serve.
  4. Tune for VRAM. Click Configure on a model to adjust what matters most: Context Size (smaller = less KV cache = less VRAM) and KV Cache Quant (q8_0 roughly halves KV memory with little quality cost). Watch the VRAM Est. column update so you can see the model fit.
  5. Start the server. Go to Server and click Start. Use the icon in the Available Models card to force-load a model into VRAM, or let the router load it on the first request.
  6. Send a request. Use the OpenAI-compatible endpoint listed in the API Endpoint card, passing a model's public name (e.g. unsloth-Qwen3.5-4B.Q4_K_M) in the model field.

Core concepts

Enabled vs loaded

These are two different things:

Enabled
The model is included in the llama.cpp preset file and advertised via /v1/models. The router knows it exists and can load it on demand. You can enable as many models as you like. Enabling does not use any VRAM.
Loaded
The model's weights are actually resident in GPU VRAM and ready to serve a request. Only a limited number can be loaded at once (Max Loaded on the Server card). The router loads a model when the first request arrives or when you click the icon in the Server tab's Available Models card.

So: "enabled" means "available to serve"; "loaded" means "actively in VRAM now".

Max Loaded

Caps how many models can be simultaneously resident. When the cap is hit and a request comes in for another enabled model, the router unloads the least-recently-used one to make room. Set to 0 for unlimited (not recommended unless you really do have the VRAM).

Public model names

The id returned by /v1/models is a short name derived from the HuggingFace repo and quant — e.g. unsloth-Qwen3-Coder-Next.UD_Q8_K_XL. The long registry ID still works as an alias. Use the copy icon in the Server tab to grab the name for your client.

Presets & aliases

Each enabled model's flags and aliases are handed to llama-server when it starts. User-defined aliases (set in Configure → Aliases) are added alongside the auto-generated ones, so you can refer to a model by any of: public name, registry ID, or your own alias.

Server tab

The single page you'll spend the most time on. It combines live status, server controls, and logs.

Server card

  • State badge — Stopped / Starting / Running / Failed.
  • Build — which compiled llama.cpp to run. Auto (newest ref) picks the successful build with the highest bN tag.
  • Max Loaded — see above.
  • Start / Stop / Restart — control the llama-server process.

Available Models card

Lists every enabled chat model by public name. Each row has:

  • A load icon — force-loads the model into VRAM. Only appears when the model is not yet loaded.
  • A copy icon — copies the public name to the clipboard.
  • A loaded / loading tag when the router has it resident.

Inventory & API Endpoint cards

Build/model counts and the OpenAI-compatible base URL for your clients.

Live Performance

Passive timing summary collected from every /v1/chat/completions request that flows through the proxy — no explicit benchmark needed. Shows rolling averages of generation and prompt-processing speed per model, so you can see real-world performance as you use the server. Both streaming and non-streaming requests are captured.

Server Logs

Live-tailing stdout/stderr from llama-server. Live tail auto-scrolls. Copy dumps to clipboard; Clear wipes the pane (does not clear backend history).

Builds

llama.cpp must be compiled against your hardware. The Builds tab checks out a ref from ggml-org/llama.cpp, configures cmake for a backend profile, builds, and keeps the resulting llama-server binary. Multiple builds can coexist — you pick the active one on the Server tab.

Profiles

CPU
Portable build with no GPU acceleration.
CUDA
NVIDIA GPUs. Requires the CUDA toolkit in the container.
ROCm
AMD GPUs. Requires ROCm. Pick an arch that matches your card (e.g. gfx1201 for Radeon R9700).

Ref selection

Choose a specific llama.cpp release tag (b*) or latest to auto-pick the newest. Some features (like experimental tensor parallelism) only exist after a certain ref.

Models

Every downloaded GGUF shows up here. The list is grouped by HuggingFace org, then by base model name (the repo minus -GGUF), with all quants of a base model shown together.

Controls

Filter box
Substring match across org, base name, quant, public name, and architecture. Matching rows auto-expand their group.
Collapse / Expand All
Toggle every org group at once. Per-base-model collapse state is preserved across org toggles.
On switch
Enable/disable the model. Disabled models are not served; restart the server to fully remove from its registry.
Configure
Expands an inline form to edit per-model flags (see below).
Remove
Removes the model from the UI. Choose "Keep Files" to leave GGUFs on disk, or "Delete Files" to also remove them.

VRAM Est. column

Shows base (weights + small overhead) and peak (weights + full KV cache at configured context size). The dashboard's GPU allocation map uses the base number because the router only ever loads a subset at once.

Model config (Configure dialog)

GPU Assignment

A single dropdown that chooses how the model is spread across your GPUs:

GPU N
Pin to a single GPU.
GPUs i-j
Contiguous range, layer split — each GPU holds a sequential slice of layers.
All GPUs (layer split)
Layer split across every GPU.
Tensor Parallelism (N GPUs)
Experimental. Splits individual tensors across N GPUs simultaneously — needs a fast GPU interconnect. Upstream support is still rough for some architectures.
Custom
You provide a raw --tensor-split ratio string.

Key flags

GPU Layers
How many layers to offload. 999 means "all"; lower values keep some on CPU.
Context Size
Token context window. Larger = more VRAM for the KV cache. "Model Default" uses the training context length.
Flash Attention
Usually safe on, faster and lower VRAM.
Jinja Templates
Required for most modern chat models to render their prompt format.
KV Cache Quant
Quantize the KV cache. q8_0 is a ~50% VRAM win with almost no quality cost; q4_0 saves more but can affect quality.
Direct I/O
Load model weights straight to VRAM, bypassing the OS page cache. Useful when VRAM > system RAM.
Vision Projector (mmproj)
Multimodal projector path. Auto-detected from the model directory for most vision models.

Speculative Decoding

Predict multiple tokens per step for a speedup with no quality change. Draft Model uses a smaller same-architecture model as the drafter (largest and most consistent speedup). N-gram modes use pattern matching with no extra model — best for repetitive workloads. Click the ? button in the Configure dialog for per-mode details.

Sampling

Temperature, top-p, top-k, min-p, penalties — applied per-request with no restart needed. Leave blank to use server defaults.

Extra Flags

Free-form llama-server flags appended verbatim. Use for anything not exposed in the UI.

Most config changes require a server restart to take effect (the flags are baked into the preset file that llama-server reads at startup). Sampling params apply per-request with no restart.

Search HF

Search HuggingFace for GGUF repos, browse files, and download specific quants. Downloads resume on interruption. Large models are shown with their total size so you know what you're committing to.

Gated models require a HuggingFace token — set it under Settings.

Vision models & projector files

Some vision-capable LLMs ship as two files: the main GGUF (text+vision weights) and a separate mmproj file (the vision projector — maps image embeddings into the model's text space). On HuggingFace these usually live in the same repo, with names like mmproj-model.gguf.

  • Downloading the main GGUF does not automatically fetch the mmproj. You need to click it in the file list to download it too.
  • Once the mmproj is downloaded into the same model directory, it's auto-associated with its LLM — the Configure dialog's Vision Projector (mmproj) field fills in automatically.
  • If the LLM has vision baked in (no separate projector needed), the Configure dialog shows Vision: Built-in and there's nothing to download.
  • mmproj files show up in the file list with a small badge but are not registered as standalone models.

If an mmproj ends up in an unexpected location, you can paste its path into the Configure dialog's Vision Projector field manually.

Benchmarks

Runs either the llama.cpp llama-bench microbenchmark or real end-to-end requests through the router against a loaded model. Results persist and can be compared across runs.

Presets

Quick (~10s)
One end-to-end request with a 256-token prompt and 128 generated tokens. Fast sanity check that a model loads and responds; does not run llama-bench.
Standard (~2 min)
Three repetitions of end-to-end requests at 128, 512, and 2048-token prompts (128 generated tokens each). Also runs llama-bench for raw prompt-processing and token-generation throughput, independent of HTTP overhead.
Thorough (~10 min)
Five repetitions at 128 / 512 / 2048 / 8192-token prompts with 256 generated tokens each, plus llama-bench. Use this when you care about long-context performance or need tight variance bounds.

What gets measured

  • PP t/s — prompt processing speed: how fast the model ingests your input. This is what determines "time waiting before streaming starts".
  • TG t/s — token generation speed: the streaming rate users actually feel.
  • TTFT — time to first token, including prompt processing. Lower = snappier perceived response.

Actions

  • Detail — expands per-run results with all data points and hardware snapshot.
  • Compare — check multiple rows and hit Compare to see them side by side.
  • Export — CSV dump of every data point for external analysis.
Note: llama-bench does not support multi-file (sharded) GGUFs. For large models split into multiple files, only the end-to-end portion of the benchmark runs; the llama-bench badge won't appear on the row.

A failed benchmark still appears in the list and can be deleted like any other.

Settings

Auto-start
Start the llama-server automatically when the container boots.
Server URL
The externally-reachable base URL of this web UI — used for the clickable API endpoint and Chat UI links. Change this if you access the UI from another host.
OpenAI-Compatible Proxy
The /v1 endpoint exposed to clients. Test Connection verifies the llama-server is reachable.
API Key
Optional Bearer token required for /v1/* requests. Leave blank for an open proxy.
HF Token
Required to download gated models.
Theme
UI color scheme. Saved per-browser in localStorage.

Using the API

The server speaks the OpenAI /v1/* dialect. Point any OpenAI-compatible client at the proxy endpoint.

curl $BASE/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "unsloth-Qwen3.5-4B.Q4_K_M",
    "messages": [{"role": "user", "content": "hello"}]
  }'

Pass any of these in the model field: public name, registry ID, auto-discovery name, or a user-defined alias. The router loads the model on first request (may take a few seconds).

Troubleshooting

Start button does nothing
No successful build exists. Go to Builds and build llama.cpp first.
"no compiled build available"
Same — the selected build failed or none exist.
Model won't load / crashes llama-server
Check the Server Logs. Tensor parallelism is experimental and may fail on some architectures. Try switching GPU Assignment to a layer-split option.
Model doesn't appear in /v1/models
Verify the "On" switch is flipped, then restart the server so the preset file regenerates.
Model still shows as "loading" forever
It likely failed at startup — check Server Logs for the error.
Out of VRAM
Lower GPU Layers, reduce Context Size, enable KV Cache Quant, or lower Max Loaded. The GPU allocation map on the Models page shows where your VRAM is going.
